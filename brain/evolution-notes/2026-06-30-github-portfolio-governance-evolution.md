# Evolution Note · GitHub Portfolio Governance

Date: 2026-06-30  
Type: evolution note  
Source Issues: #148 / #149 / #150 / #151 / #152 / #153 / #154  
Writeback Level: L3 evolution writeback

## 1. Previous State

Before this evolution, GitHub governance was issue-centered and project-centered:

```text
single repo
→ issue
→ C3 Gap / C4 Task
→ evidence / evolution
```

This worked for `yuanli-strategy-soul`, but became insufficient once the repo portfolio expanded across:

```text
strategy canon
C2 brain
content engineering
client/company brain
investment/research
engineering/tooling
agency/skill repos
personal OS / experiments
```

## 2. New Realization

The portfolio itself has become a strategic object.

```text
GitHub is no longer a code storage layer.
GitHub is becoming the strategic capability registry of Yuanli OS.
```

Therefore, the governance unit must evolve:

```text
from repo-level PMO
→ to portfolio-level CBM-PMO
```

## 3. New Operating Principle

```text
Govern capability, not repo noise.
```

This means:

```text
Repo activity is not value.
Repo value = CBM Cell + Evidence + Next C3/C4 + Writeback Target + Reuse Path.
```

## 4. New Structure

The governance system now uses three layers:

```text
L1 Core CBM-PMO Governance
L2 Capability Asset Pool
L3 Observe / Archive Candidate
```

This prevents two opposite errors:

1. Under-governance: valuable repo activity disappears into GitHub noise.
2. Over-governance: every small tool/demo/agent repo pollutes the main PMO.

## 5. Canonical Impact

This evolution strengthens C3 / C4:

```text
C3 一张地图
→ sees the capability portfolio, not just isolated projects.

C4 一条链路
→ turns repo activity into C3 Gap, C4 Task, Evidence, Evolution, and Writeback.
```

It also strengthens C2:

```text
C2 一个大脑
→ gains a cleaner source boundary between canon, evidence, tool, client, and content repos.
```

## 6. New Policy Concepts

The following concepts should become reusable PMO language:

```text
L1 Core Governance
L2 Capability Asset Pool
L3 Observe / Archive Candidate
False-green Risk
Evidence Strength
Repo Intake Rubric
Human-gated Archive / Merge / Rename
Called-by Project
Reuse Contract
Portfolio Review Rhythm
```

## 7. Current Canonical Candidate

Candidate learning:

```text
A GitHub repo should not be governed because it exists or because it is active.
It should be governed only when it represents a reusable strategic capability with evidence, next action, and writeback path.
```

Status:

```text
Canonical Candidate only.
Not canon approved.
Needs application through #153 / #154 / #155 / #156 / #157.
```

## 8. Next Evolution Trigger

This evolution can be upgraded after:

```text
1. At least 80 six-month active repos are classified.
2. Top 15 L1 repos are mapped into CBM-PMO.
3. At least 30 capability assets receive reuse contracts.

```

## 9. Boundary

This evolution does not authorize:

```text
archive
merge
delete
rename
visibility change
maturity upgrade
canon upgrade
```

All such actions remain human-gated.
