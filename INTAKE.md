# INTAKE · 原力衍生任务唯一入口规则

> 本文件定义 `yuanli-strategy-soul` 如何作为所有原力战略三部曲衍生任务的入口仓。

---

## 1. 总规则

任何新任务，先进入本仓 Issue，不直接进入执行仓。

```text
想法 / 需求 / 项目 / Bug / 页面 / Agent / Codex任务
→ yuanli-strategy-soul Issue
→ 归属裁决
→ 任务拆解
→ 分发执行
→ 回写沉淀
```

---

## 2. 为什么必须先进入本仓

因为原力战略三部曲不是任务清单，而是 ʸx 系统。

如果一个任务没有先裁决：

```text
属于哪一部？
服务哪条脊骨？
输出物是什么？
回写到哪里？
```

它很容易变成一次性劳动，无法复用、无法沉淀、无法递归。

---

## 3. Issue 类型

| 类型 | 模板 | 用途 |
|---|---|---|
| Universal Intake | `00-universal-intake.md` | 所有新想法、新需求、新任务的默认入口 |
| Concept Governance | `01-concept-governance.md` | 概念、命名、边界、正典裁决 |
| Codex Task | `02-codex-task.md` | 可以交给 Codex 执行的明确任务 |
| Derived Project | `03-derived-project.md` | 网站、产品、Agent、看板、课程等衍生项目 |

---

## 4. 分发规则

### 4.1 留在本仓

以下任务留在本仓处理：

```text
概念宪法
任务治理
Issue 模板
Prompt Pack
看板模板
入口网站
跨仓路线图
```

### 4.2 分发到内容仓

以下任务分发到内容 / 正文 / 课程仓：

```text
章节写作
课程大纲
公众号文章
概念长文
案例库整理
```

### 4.3 分发到网站仓

以下任务分发到网站 / 前端 / 展示仓：

```text
HTML 页面
交互网站
可视化座舱
知识地图
产品原型
```

### 4.4 分发到 Codex 执行仓

以下任务可交给 Codex：

```text
代码实现
文件重构
组件开发
脚本生成
自动化流程
测试与修复
```

但必须先有明确的：

```text
输入 / 输出 / 验收 / 禁止事项 / 回写位置
```

---

## 5. 归属裁决

每个 Issue 必须选择一个主归属：

```text
原力资产：找到你
原力创业：变现你
原力OS：系统你
跨三部：影响多条主线
治理：影响仓库规则、模板、看板、路线图
```

---

## 6. 脊骨裁决

每个 Issue 至少挂回一条共享脊骨：

```text
ʸx
你 × AI
借势 × 合力
四关
信息五层 / OSA
递归回写
```

不能挂回任何脊骨的任务，默认不进入核心任务池。

---

## 7. 回写规则

每个任务完成后必须选择至少一个回写位置：

```text
CONSTITUTION.md
README.md
docs/
codex/
project/
.github/ISSUE_TEMPLATE/
具体执行仓
知识库 / 大脑
```

---

## 8. Done 标准

一个任务只有满足以下条件才算 Done：

```text
1. 输出物已交付
2. 验收标准已满足
3. 回写位置已更新或说明
4. 下次复用方式已明确
5. 若需分发，已标明目标仓库
```

否则只能进入 Review 或 Backwrite。

---

## 9. 标准入口文案

以后任何新任务可以这样开头：

```text
请在 yuanli-strategy-soul 中建立一个入口 Issue：
任务名：
背景：
初步归属：资产 / 创业 / OS / 跨三部 / 治理
预期输出：
是否适合 Codex：是 / 否 / 待判断
```
